# /brainstorm — 設計の壁打ち → spec を docs/specs/ に残す

アイデアを、対話を通じて実装可能な設計(spec)に落とし込む。superpowers プラグインに依存しない、このリポジトリの自己完結版。

## HARD-GATE

設計をユーザーに提示し承認を得るまで、実装・コード生成・ファイルのスキャフォールドをしない。「単純すぎて設計不要」と判断しない — 短くてよいので必ず設計を提示し承認を取る。

## フロー

1. **文脈の調査**: 関連ファイル・`docs/`・直近コミットを見て現状を把握する。`AGENTS.md` のドキュメントマップから関連ドキュメントを辿る。
2. **要件を 1 問ずつ確認**: 多肢選択を優先し、1 メッセージ 1 問。purpose / 制約 / 成功基準を詰める。明確になるまで実装の話に進まない。スコープが大きすぎる(複数の独立サブシステムを含む)場合は、まず分解して最初の 1 つを設計する。
3. **アプローチを 2-3 案**: トレードオフと推奨を添えて提示する。推奨を先頭に、理由を述べる。
4. **設計をセクションごとに提示**: 複雑さに応じた分量で。各セクションごとに「ここまで合っているか」を確認する。アーキテクチャ / コンポーネント / データフロー / エラー処理 / 検証 を網羅。
5. **spec を保存**: 確定した設計を `docs/specs/YYYY-MM-DD-<topic>-design.md` に書く(日付は `date +%F`)。why を書き、コードから読める what は書かない。1 ファイル 200 行以下(超えたら分割)。関連する ADR があればリンクする。
6. **自己レビュー**: プレースホルダ / 矛盾 / スコープ過大 / 曖昧さ を点検し、その場で直す。
7. **ユーザーレビュー**: spec のパスを伝え、レビューを依頼する。修正要望があれば直して再点検。
8. **実装へ**: 承認後に実装へ入る(実装計画のスキルやサブエージェントがあれば使う)。

## 原則

- YAGNI。不要な機能は設計から削る。
- 既存コードベースでは、まず現状の構造を調べ、既存パターンに合わせる。無関係なリファクタは提案に留める。
- このリポジトリの規約は `AGENTS.md`(と、そこからリンクされる `docs/`)に従う。コミット / push は明示依頼時のみ。
